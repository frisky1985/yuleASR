# 设计文档

## 架构决策

### 测试框架设计

采用自研轻量级测试框架，而非集成第三方框架（如 CMocka/Unity），原因：
1. 减少外部依赖
2. 更好的定制化能力
3. 更轻量的代码库
4. 更容易集成到现有工具链

### 测试分层

```
tests/
├── unit/           # 单元测试 (针对单个模块)
├── integration/    # 集成测试 (模块间交互)
├── mocks/          # 模拟对象
└── tools/          # 测试工具
```

### Mock 策略

使用链式 Mock 机制，支持：
- 函数调用记录
- 返回值设置
- 错误注入
- 调用次数验证

## 技术选型

| 组件 | 选型 | 理由 |
|------|------|-------|
| 测试框架 | 自研 | 轻量、定制化 |
| 断言系统 | 宏定义 | 清晰、可维护 |
| Mock 系统 | 链式 | 灵活、可扩展 |
| 报告生成 | Python | 丰富的库支持 |
| CI/CD | GitHub Actions | 无需额外基础设施 |

## 质量保障

1. 所有测试用例独立可运行
2. 每个测试用例有明确的断言
3. Mock 对象支持错误注入
4. 测试报告支持多格式 (HTML/Text/JSON)
